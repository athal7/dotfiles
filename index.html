Three approval gates the agent can't skip. An opencode plugin hooks every tool call: editing files, running git commit, running git push — each is blocked until the user clicks "Approved" on a question the agent itself raised. The agent can call the question tool, but only the user can pick the answer.

Highlight

MCP → skills

My earlier setup wrapped every data source in a persistent MCP server. Curating, maintaining, keeping wrappers in sync with APIs that keep changing.

The MCP tax

A server process per source. Wrappers go stale on API drift — two surfaces to maintain. Tool descriptions sit in context every session, whether the task uses them or not.

Skills point at APIs directly

Plain markdown. The agent calls documented CLIs and REST/GraphQL endpoints. When an API changes, update two lines — no wrapper to redeploy, no schema to keep in sync. Less context per call, no server to babysit.

MCPs that stayed

context7 for live library docs, firefox-devtools for QA. Both expose capabilities a CLI call can't replicate. That's the bar.

Highlight

attention

What needs my focus right now? Pulled from across my work and life, calibrated to my energy.

Pulled in parallel

Calendar runway, reminders (overdue / today / undated), merge requests, issues — assigned and cross-referenced.

Output tuned to energy

Cognitive, social, sensory — multidimensional. Low → 1 work + 1 personal. Medium → 2–3 quick wins. High → longer-horizon work.

Highlight

conversations

Research people, decisions, and context across four sources. Each has different fidelity and recency — the skill picks in order.

Knowledge base distilled profiles — start here, fast and structured
Meetings transcripts when KB doesn't have enough detail
Chat live and undigested — today, yesterday, threads not yet ingested
Email formal correspondence, tickets, procurement

Skill is provider-agnostic: KB path, meetings CLI, chat API, email CLI all swap independently. Same skill, different stacks.

Highlight

context-log

A markdown file in the project that survives compaction and session handoffs. .opencode/context-log.md, gitignored.

Session start create with issue context — title, acceptance criteria, key references
After each commit append a checkpoint — intent, tests, what's next
On compaction don't re-summarize — point at the log instead

Compaction normally collapses everything older than the recent window into a summary. The log is the durable record of why and what's next — outside the model's memory.

Highlight

chezmoi

Skills, config, binaries, services — one chezmoi apply on any machine. Every change is a git commit.

What ships

All 26 skills → ~/.agents/skills/. opencode.json for MCPs, model, permissions. AGENTS.md.tmpl for agent instructions. Plugins, LaunchAgents, packages.

Why not OpenClaw

Chat-native, proactive, self-modifying — compelling. But my skills and config belong in git, not in an agent's own memory. Reproducible, auditable, portable.

Mobile, without a separate app

opencode web runs as a LaunchAgent on my desktop, exposed over Tailscale, installed as a PWA on my phone.
